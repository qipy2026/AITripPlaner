
# AITripPlaner 产品全流程文档

**项目名称**: AITripPlaner 智能旅行助手  
**本文档按产品经理标准工作流组织，覆盖发现→需求→设计→PRD→开发与运维**

---

## 1. 发现与调研阶段

**目标**：验证问题真实性与商业/用户价值

### 1.1 市场背景与问题定义
- **用户行为变化**：自由行占比持续上升，用户习惯自主规划行程，但制作一份周全的旅行攻略平均耗时 3-8 小时，涉及查景点、看天气、找酒店、排路线。
- **现有方案痛点**：
  - 论坛/笔记类（马蜂窝、小红书）信息零散，整合成本高。
  - 地图工具（高德、百度）可查 POI，但无行程串联能力。
  - 行程规划 App（穷游行程助手）以手工拖拽为主，无智能生成。
- **核心问题**：自由行用户缺少一种“输入偏好即可自动生成每日行程”的高效工具，尤其在需要结合实时天气和地理可达性时。

### 1.2 用户调研摘要（推断性）
- **目标用户**：25-45 岁一二线城市自由行群体，具备基础互联网操作能力，每年旅行 1-3 次。
- **典型痛点**：
  - “景点和酒店距离太远，排完才发现每天在路上 4 小时。”
  - “想找个带娃的轻松路线，结果全是网红打卡点。”
  - “天气一变，之前做的攻略全废了。”
- **核心诉求**：快（生成快）、准（行程合理）、可看（地图可视化）、可调（生成后可修改）。

### 1.3 竞品分析
| 竞品 | 优势 | 劣势 |
|------|------|------|
| 马蜂窝/穷游行程助手 | 用户基数大，模板丰富 | 纯手工拖拽，无 AI 生成 |
| 小红书攻略 | 真实体验，细节丰富 | 非结构化，无法一键成行 |
| ChatGPT 直接生成 | 文本流畅 | 不含真实地理数据，易出现“幻觉酒店” |
| 高德/百度地图 | 地点数据准，路网完善 | 无行程编排逻辑 |

**差异化机会**：将 LLM 的推理能力与高德真实 POI/路线/天气数据结合，以工作流保证输出可靠，并辅以规则兜底。

## 2. 需求定义阶段

**目标**：明确要解决哪些问题，划定范围与优先级

### 2.1 用户故事
| 编号 | 角色 | 需求 | 目的 |
|------|------|------|------|
| US-01 | 旅行者 | 输入目的地、日期、偏好，一键生成每日行程 | 快速获得可靠计划 |
| US-02 | 细节控 | 在地图上查看景点与酒店位置，并可查询路线 | 评估行程空间合理性 |
| US-03 | 天气敏感者 | 查看出行期间天气，并在行程中显示 | 动态调整户外安排 |
| US-04 | 体验尝鲜者 | 了解行程生成逻辑，失败时也能得到基本方案 | 信任系统稳定性 |

### 2.2 功能清单与优先级
| 优先级 | 功能 | 对应故事 |
|--------|------|----------|
| P0 | 行程生成（含数据采集+LLM合成+规则兜底） | US-01 |
| P0 | 每日行程卡片展示（含景点、餐饮、住宿） | US-01 |
| P1 | 高德地图标记与路线规划 | US-02 |
| P1 | 天气展示（总览及每日） | US-03 |
| P2 | 历史计划保存与加载 | - |
| P2 | 行程导出为 PDF/图片 | - |

### 2.3 关键性能需求
- 行程生成响应时间 ≤ 15 秒
- LLM 生成成功率 ≥ 90%（兜底方案 100% 可用）
- 地图首屏加载 ≤ 3 秒

## 3. 产品设计阶段

**目标**：定义交互流程、信息架构与关键原型要点

### 3.1 核心用户旅程
```
[首页] → 填写旅行需求表单（城市/日期/天数/风格）
→ 点击“生成旅行计划”
→ 过渡态：步骤提示（“正在搜集景点…”“智能编排中…”）
→ [结果页]：
    - 左侧：可按日切换的时间线卡片（景点→午餐→景点→酒店）
    - 右侧：高德地图全屏视图，联动显示当日位置
→ 用户可点击两地点查询路线（弹窗显示距离/时长）
→ 可查看天气趋势面板
```

### 3.2 界面原型关键点
- **输入表单**：卡片式布局，风格标签以可多选 Tag 呈现；必填项未完成时按钮禁用。
- **结果页**：
  - 日期切换条（横向滚动），高亮当前日。
  - 行程卡片内联景点、餐饮、住宿信息，含简短描述与标签。
  - 地图标记点支持 InfoWindow（名称、地址）。
- **响应式**：移动端下地图与卡片上下堆叠，而非左右布局。

### 3.3 数据结构草案
前端行程 JSON 结构（与后端对齐）：
```typescript
interface TripPlan {
  city: string;
  start_date: string;
  days: DayPlan[];
}
interface DayPlan {
  date: string;
  weather: { condition: string; temp: number; humidity: number };
  activities: Activity[];
  accommodation: Hotel;
}
interface Activity {
  type: 'attraction' | 'restaurant' | 'transport';
  name: string;
  location: { lng: number; lat: number };
  time_slot: string;
  duration_min: number;
}
interface Hotel {
  name: string;
  location: { lng: number; lat: number };
  price_range: string;
}
```

## 4. PRD 阶段（产品需求文档）

> 以下为标准化 PRD 精要，对应可评审、可执行的开发依据。

### 4.1 功能需求详述

#### FR-01 用户输入采集
- 输入字段：城市（必填）、日期（必填）、天数（必填）、交通偏好（多选）、住宿偏好（单选）、旅行风格（多选）
- 校验：日期不可为过去；天数 ≥1 且 ≤30；城市名称需匹配高德城市编码映射

#### FR-02 LangGraph 工作流
- 节点1 `gather_data`：并行调用高德 REST API，获取景点、酒店、天气数据，去重与缓存。
- 节点2 `llm_plan`：组装 Prompt，调用 OpenAI 兼容 LLM，要求返回严格 JSON。解析并校验字段完整性；失败则路由至 `fallback_plan`。
- 节点3 `fallback_plan`：基于 POI 热度、地理位置就近原则、天数生成每日基础行程，确保可用。
- 数据获取服务：`AmapRestClient` 封装 maps_text_search, maps_weather 等。
- LLM 调用：通过 LangChain 的 `ChatOpenAI`，配置 base_url、api_key、model name。

#### FR-03 行程展示
- 每日卡片：含日期、天气图标+温度、时间线活动列表。
- 餐饮推荐：LLM 需在每日行程中嵌入至少 2 餐推荐。
- 住宿：每日结束前推荐一家酒店（可跨日相同）。

#### FR-04 地图交互
- 使用高德 JS API 2.0 加载地图，按日标记地点。
- 点击任意两个标记点可调用 `POST /api/map/route` 获取驾车/步行/公交路线并绘制。
- 地图自动适配视野，使当日所有标记点可见。

#### FR-05 天气总览
- 以条形图或横向滑动卡片展示行程期间每日天气概要。

### 4.2 非功能需求
- **安全**：API Key 仅存储于环境变量，前端不可见；输入防 XSS。
- **可维护性**：后端模块化（agents/services/models/api），前端 TypeScript 覆盖主要类型。
- **可扩展性**：LLM 提供商可通过 `.env` 切换，地图服务可替换。
- **监控**：记录每次生成耗时、LLM 成功率、高德 API 调用次数。

### 4.3 API 端点定义
| 方法 | 路径 | 描述 | 请求/响应关键字段 |
|------|------|------|------------------|
| POST | `/api/trip/plan` | 生成旅行计划 | 入参: `TripRequest` (city, start_date, days, preferences)；返回: `TripPlan` |
| GET | `/api/map/poi` | 搜索 POI | 参数: city, keywords, type；返回: POI 列表 |
| GET | `/api/map/weather` | 查询天气 | 参数: city；返回: 每日天气对象数组 |
| POST | `/api/map/route` | 规划路线 | 入参: origin, destination, mode；返回: route 对象（距离、时间、路径坐标） |

### 4.4 验收标准
- [ ] 输入错误时前端有明确提示，不允许提交非法值。
- [ ] 正常流程下 10 天内行程均能成功生成，JSON 结构与 schema 一致。
- [ ] 故意造成 LLM 超时后，系统可自动进入兜底方案并返回可用行程。
- [ ] 地图标记点与返回数据坐标一致，路线查询返回合理的距离和时间。
- [ ] 页面在 375px 宽度（iPhone 6/7/8）下无横向滚动条，布局正常。

## 5. 开发与迭代计划

### 5.1 里程碑
| 里程碑 | 完成内容 | 预计产出 |
|--------|----------|----------|
| M1 | 后端工作流与基础 API 调通 | 可返回 mock 行程 JSON |
| M2 | 前后端联调，核心生成闭环 | 真数据可生成并在控制台打印 |
| M3 | 前端结果页与地图集成 | 可视化展示，地图标记与路线 |
| M4 | 异常处理与兜底方案实现 | LLM 失败时降级可用 |
| M5 | 优化与发布 | 性能达标，文档完善，上线 |

### 5.2 风险应对
- LLM 输出格式不稳定 → 强化 Prompt + Pydantic 校验 + fallback
- 高德 API 限流 → 本地缓存常用城市数据，限流时提示用户稍后再试
- 长行程（>7天）生成慢 → 拆分多批调用或采用流式返回
